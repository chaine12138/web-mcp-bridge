
## 一、定位

**Web-MCP-Bridge 不是浏览器自动化**，而是：

> **把网页里业务系统自己暴露的 API，通过 MCP 协议反向映射给 AI Agent。**

类比：宿主网页（低代码平台、CRM、后台系统…）就像一台"浏览器里的服务器"，它本来只给页面自己的 UI 用；我们做的是给它装一根**反向网线**，让 Agent 能从 IDE 端直接调它的能力。

本质是一个 **"应用内 SDK ↔ MCP" 的双向 RPC 桥**，浏览器只是宿主进程。

---

## 二、架构

```
┌──────────┐  MCP/stdio  ┌──────────────┐   WebSocket   ┌────────────────────────┐
│ AI Agent │ ──────────▶│ Local Bridge  │ ◀───────────▶│  宿主 Web 应用          │
│          │            │               │               │  ├─ web-mcp-sdk (npm) │
│          │ ◀─tools──  │ (动态 tool    │               │  └─ 业务注册:         │
└──────────┘            │  registry)    │               │     updateComponent    │
                        └──────────────┘               │     deleteComponent    │
                                                       │     addComponent ...   │
                                                       └────────────────────────┘
```

三层关键变化：

1. **浏览器端不是扩展**，而是一个 **`web-mcp-sdk` npm 包**，由宿主应用主动引入（低代码平台自己 `import` 它）。
2. **工具列表是动态的**：Agent 连上后，先向宿主应用发 `list_tools`，宿主返回自己注册过的业务能力，再由 Bridge 包装成 MCP tools 暴露给 Agent。
3. **工具实现在宿主应用里**：Bridge 只是透传，不关心业务；`updateComponent` 的真实逻辑是宿主平台自己的代码。

## 三、核心协议（SDK → 宿主开发者的使用姿势）

宿主方（比如低代码团队）接入只需几行：

```ts
import { createMcpBridge } from 'web-mcp-sdk';

const bridge = createMcpBridge({
  appId: 'lowcode-studio',
  endpoint: 'ws://127.0.0.1:7321',
  token: 'xxx',
});

bridge.register({
  name: 'updateComponent',
  description: '修改指定 id 的组件属性',
  inputSchema: z.object({
    componentId: z.string(),
    props: z.record(z.any()),
  }),
  handler: async ({ componentId, props }) => {
    return store.updateComponent(componentId, props); // 调自己的业务
  },
});

bridge.register({ name: 'deleteComponent', /* ... */ });
bridge.register({ name: 'addComponent',    /* ... */ });

bridge.connect();
```

对 Agent 来说，看到的 MCP 工具就是 `lowcode-studio.updateComponent` / `lowcode-studio.deleteComponent`，**工具名和 schema 完全由宿主自己定义**。

## 四、Bridge 本地服务的职责（薄一层）

| 职责 | 做什么 |
|---|---|
| MCP Server | 向 Agent 暴露标准 MCP；`ListTools` 动态聚合所有已连接的宿主页面工具 |
| WS Server | 监听 `127.0.0.1:<port>`，接受多个宿主会话（可能同时有多个 Tab） |
| 会话注册表 | `appId + tabId → 工具集合` 的映射，支持多应用并存 |
| 调用路由 | Agent 调 `lowcode-studio.updateComponent` → 定位到对应 WS → 透传 → 等待回包 |
| 工具命名空间 | 用 `appId.toolName` 防冲突；多 Tab 时可加 `instanceId` |
| 生命周期 | Tab 关闭自动下线工具；断线重连；请求超时 |

Bridge **不实现任何业务工具**，是纯粹的协议桥。

## 五、关键能力边界（和之前版本的对比）

| 能力 | 之前（浏览器自动化版） | 现在（业务 API 桥版） |
|---|---|---|
| 定位元素 | CSS selector / a11y tree | 不需要，按业务 id |
| 点击/输入 | 内置工具 | ❌ 不提供（宿主不需要） |
| 截图/导航 | 内置工具 | ⚠️ 可选作为辅助 |
| 业务操作 | Agent 自己组合点击 | 宿主一行注册，Agent 直接调 |
| 接入方式 | 装扩展 | 宿主 `npm i` 即可 |
| 通用性 | 任意网站 | 仅接入了 SDK 的网站 |
| 稳定性 | 依赖 DOM 结构，易断 | 业务 API 级稳定 |

**结论**：这是一个**让 Web 应用"MCP 化"的 SDK**，而不是一个浏览器控制器。

## 六、建议的 MVP 范围

**Bridge（Node/TS）**
- MCP server（stdio） + WS server（127.0.0.1）
- 多会话 / 多应用工具聚合
- 命名空间 + 超时 + 断线重连
- 一条 `health_check` 内置工具，用于 Agent 探活

**SDK（浏览器 TS）**
- `createMcpBridge` / `register` / `connect` / `disconnect`
- Zod schema 定义工具入参
- 自动重连、心跳、错误序列化
- 可选：`registerResource`（暴露只读数据，如"当前画布 JSON"）

**示例宿主**
- 一个极简低代码 Demo（画布 + 3 种组件），演示增/删/改 + 保存

**Agent 验收**
- 用 Qoder 跑："在画布里新增一个按钮，文案改成'提交'，并删除 id=header 的组件" → 能一次完成
